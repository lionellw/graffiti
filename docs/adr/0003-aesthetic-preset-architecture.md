# Aesthetic preset architecture: opt-in classes, token-first, layered between base and components

Graffiti will ship "aesthetic presets" — named CSS classes such as `.theme-editorial`, `.theme-brutalist`, `.theme-soft-consumer`, `.theme-neon-arcade`, `.theme-paper` — that bundle a coordinated stack of CSS-variable overrides (and, sparingly, base-element selector rules) to deliver a recognisable visual personality with one class application. Presets are distributed via a separate opt-in import path, not bundled into `drop-in.css`, and their selector rules live in a new `@layer themes` slotted between `base` and `components`. The overall cascade declaration becomes `@layer base, themes, components, utilities, layouts;` ([ADR-0002](./0002-cascade-layer-order.md) was extended, not overridden).

The driver is `ThemeControls.svelte`, which already exposes color, type-scale, radius, and font-family axes orthogonally. Consumers (and AI agents using the framework) can mix selections freely, but the result rarely looks "transformed" — orthogonal axes produce variations of the same look, not different looks. An aesthetic preset is a non-orthogonal artifact that couples decisions across axes to deliver look quality on its own. The same class can be applied at `:root` / `html` for app-wide effect or on any container element for scoped theming; token overrides cascade naturally, and pseudo-element rules (`::selection`, `::first-letter`) scope to descendants automatically.

Presets are token-dominant by design: ~95% of a preset is variable overrides, and selector rules are reserved for typographic decoration (uppercase headings, drop caps, selection styling) that has no token surface today. When a preset author finds themselves reaching for a selector rule, that is treated as a signal that the framework's token surface should expand — the next iteration adds the missing token rather than embedding the selector in more presets.

## Considered options

- Skill-only documented recipes, no framework artifact. Rejected: every consumer reimplements the override stack; AI agents and humans can't share a definition of "Brutalist"; presets become invisible to anyone not using the skill.
- CSS classes shipped inside the main `drop-in.css` bundle. Rejected: every consumer pays the bytes whether they use presets or not, and the bundle grows linearly with the catalog. Aesthetic presets are opinionated; opting in must be an explicit consumer choice.
- An enforced selector allowlist with build-time validation. Rejected as too intense: preset authors are humans who should retain agency. The framework should steer toward the right path, not police it. Cascade placement and token-first conventions handle the steering.
- Themes layer slotted higher (between `components` and `utilities`, or at the top). Rejected: would let themes outrank component or layout contracts and break structural primitives. Counter to the project's "system integrity outranks aesthetic expressiveness" stance.
- Per-aesthetic separate npm packages (`@drop-in/theme-editorial`, etc.). Rejected: install/import friction is disproportionate for opt-in features that ship at the same release cadence as the core. A single shared package path with multiple entry points keeps versioning aligned and discovery simple.

## Consequences

- Presets are authored primarily as `:root.theme-x` / `.theme-x { --token: value; }` blocks. Selector rules are permitted but treated as an escape hatch.
- Because `@layer themes` sits below `@layer components`, a preset cannot accidentally restyle `.card`, `.button`, or `.dialog` internals — it can only re-skin them via the component override tokens those components already expose (`--button-color`, `--button-bg-top/bottom`, `--tag-color`, etc.). When a preset wants behavior the existing component tokens can't deliver, the framework gains a new component token rather than the preset gaining a new selector. Over time the token surface densifies and the need for preset-side selector rules decreases.
- A preset can be applied to any container, not just `:root`. Token overrides cascade to descendants; descendant-selector rules scope automatically; `color-scheme` declarations on the preset class compose correctly with `light-dark()` resolutions inside the scoped subtree.
- The font roster in `ThemeControls.svelte` must expand before the `theme-editorial` and `theme-paper` presets are deliverable — current options are all sans-serif or mono, and serif/display-condensed faces are prerequisites for the editorial and print-feel territories.
- The cascade-layer declaration in `drop-in.css` becomes `@layer base, themes, components, utilities, layouts;`. The `themes` layer is declared in the core stylesheet so the order is stable even when no preset import is present, and populated only when a consumer imports the preset package.
- The `graffiti-best-practices` skill gains a reference file describing each shipped preset's intent and a workflow step: "when the user requests a particular feel, apply a preset class first, then override tokens for fine-tuning." The skill's existing "do not invent tokens" rule remains intact — presets are not invention, they are documented surface.
- Consumer CSS (unlayered) still wins over preset selectors, so consumers retain final control without having to defeat the preset layer explicitly.
